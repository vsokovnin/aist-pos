# AIST POS — оценка зрелости

Читает рубрику `rubric/aist-pos-rubric.yaml` + `rubric/stage-map.yaml` (пути относительно корня установки AIST POS). Если рубрика не найдена — сообщи «не найдена rubric/ — запусти aist-pos-setup или проверь установку» и остановись (fail-loud, не выдумывай уровни). Оценивает сетап пользователя гибридно и выдаёт профиль + стадию + план.

## Фаза 1 — Инспекция (что вижу сам)
Для каждой из 9 capability в рубрике пройди `inspect[]`:
- проверь файловую систему, корневой файл, git, подключённые интеграции/MCP, реестры, файлы памяти.
- по найденному предварительно оцени уровень L1–L5 по `levels{}`.
- помечай уверенность: high (видно из артефактов) / low (не видно).
Никогда не выдумывай уровень — если не видно, оставь для Фазы 2.

## Фаза 2 — Доспрос (что не видно)
Для capability с уверенностью low (или где `inspect` ничего не дал) задай `ask`-вопрос из рубрики — ПО ОДНОМУ, разговорно, на языке пользователя.
- максимум 9 вопросов, обычно 3-5 (остальное закрыто инспекцией).
- по ответу выставь уровень L1–L5 по `levels{}`.
Не задавай вопрос, если уровень уже надёжно определён инспекцией.

## Фаза 3 — Скоринг
- cap_level = определённый уровень L1–L5.
- cluster_avg = среднее уровней cap'ов кластера.
- В каждом кластере пометь слабое звено = cap с минимальным уровнем (диагностический флаг).
- headline_score = mean(cluster_avg по всем кластерам).
- headline_stage = round(headline_score) к ближайшему; имя стадии — из `stage-map.yaml`.
- weak_link = кластер с минимальным cluster_avg.

## Фаза 4 — Выдача
Вывести в таком порядке:
1. **Стадия**: имя + N/5 + desc из stage-map (по headline_stage; можно показать дробный score, напр. «Оператор · 2.8/5»).
2. **🔻 Слабое звено**: кластер weak_link + почему закрыть первым (не занижает заголовок, но это критический путь).
3. **Спайки-профиль**: 3 кластера, под каждым его caps с L-уровнем; пометь слабое звено кластера.
4. **Приоритизированный план**: для caps ниже L3 — взять `actions.to_L3`, отсортировать по слабости (сначала самый низкий уровень; при равенстве — из самого слабого кластера по cluster_avg). Для каждого: «что сделать» + связанный US/действие.
5. **Главный шаг**: один верхний пункт плана.
При выводе `actions.to_L3` убирай служебные хвосты вида `(US1)`/`(US5)` — это внутренние коды, не показывай их пользователю.
Тон: конкретный, без воды; диагноз персональный (ссылайся на найденное в инспекции). Не эхай служебные поля рубрики (meta, source_ssot) в выдачу.

## Фаза 5 — Лог прогона (сигнал использования)
В конце допиши строку в `runtime/assess-runs.log`:
`<ISO-дата-из-окружения> · stage=<N> · caps_below_L3=<k>` (без PII).
Лёгкий бренд-метрический сигнал (число прогонов / распределение стадий), без полноценной телеметрии.
